Computational Inequalities · Final Storyboard

When Missed Dailies Become Damage

This project examines Habitica’s Dailies system as a computational model that turns interrupted participation into measurable loss. Rather than treating inequality as a single biased output, the storyboard traces how it is produced across the pipeline: through data structure, task taxonomy, rule-based interpretation, interface feedback, and a small intervention designed to make that logic less punitive and more legible.

Panel 1

The system and the inequality

Habitica is not just a to-do list. It turns tasks into a game system where repeated participation is linked to visible user status. In this project, the inequality is defined as the punitive translation of interrupted participation into damage, loss, and failure signals.

Evidence: dashboard screenshot showing the platform as a combined task, avatar, and progress system.

Panel 2

How routine becomes task data

Habitica does not record everyday life in its full complexity. It converts routine into structured task data: a title, a task type, a repeat pattern, a completion state, and a measurable effect. This process makes routine actionable for the system, but it also narrows what can be expressed.

The interface can register whether a Daily was completed, but it does not record why a task was missed. Fatigue, care work, unstable connectivity, emotional difficulty, or intentional rest do not appear as first-class fields. In this sense, absence is not neutral missing information. It is a product of how the system defines what counts as relevant data.

Evidence: the Daily editing interface and the project CSV show how lived routine is translated into structured fields, while many contextual conditions remain outside the record.

Panel 3

Taxonomy and the limits of legibility

Habitica organises activity through a taxonomy of Habits, Dailies, To-Dos, and Rewards. This classification system does more than sort tasks. It assigns different temporal expectations, different forms of value, and different consequences for non-completion.

Once an activity is classified as a Daily, it becomes part of a continuity logic. It is no longer just something the user hopes to do. It becomes a repeated obligation that the system expects to be maintained across time. This means that taxonomy is not only descriptive. It actively shapes how participation is measured and how interruption is interpreted.

Evidence: the task type interface and the project taxonomy reading show that Dailies are governed differently from To-Dos or Rewards, making continuity a specific condition of system legibility.

Task taxonomy summary

  • Habits: flexible repeated actions
  • Dailies: recurring tasks tied to continuity
  • To-Dos: one-off tasks with deadline logic
  • Rewards: optional self-awarded outcomes

Panel 4

Interrupted participation becomes computable damage

In Habitica, a missed Daily does not remain an unrecorded gap. It becomes a negative system event. Once repetition has been defined in advance, failure to complete a cycle can be translated into measurable loss, such as damage to health points or a decline in visible progress.

This matters because the system does not evaluate interruption in context. It evaluates whether a required action was completed within the expected timeframe. Different lived conditions can therefore produce the same negative outcome. In this way, interruption is not only noticed by the system. It is operationalised as damage.

Panel 5

The interface returns interruption as personal failure

Habitica does not only calculate task outcomes in the background. It makes them visible and emotionally legible through the interface. Health bars, avatar condition, and task feedback turn missed participation into something the user can see and feel as decline.

This is where system logic becomes affective. What may begin as unstable routine, care pressure, exhaustion, or disconnection is returned through the interface as loss, damage, or reduced status. The structural conditions behind interruption are not shown directly. Instead, the user encounters a personalised sign of failure.

Evidence: the avatar and HP display, together with the annotated screenshot, show how the interface translates a rule-based calculation into a visible and personal experience of decline.
